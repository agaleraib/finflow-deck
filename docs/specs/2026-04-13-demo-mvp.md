## 1. Overview

### What is this

A sales-ready demo application that walks a prospective client through the complete FinFlow content pipeline: from news event detection through content generation, human approval, translation, compliance review, and publish-ready output. Alex drives the demo; the prospect watches.

### Who is it for

B2B forex/finance content clients (brokers, asset managers, fintech firms). The demo proves FinFlow's end-to-end value proposition: automated market intelligence that respects brand voice, compliance rules, and editorial standards -- with human oversight at critical checkpoints.

### What problem does it solve

Prospects need to see the full pipeline in action -- not a slideshow, not a prototype. They need to see their own brand profile extracted from their samples, their glossary applied, their compliance rules enforced, and their content generated in real time. This demo is the closer.

### Product narrative (the story the demo tells)

The demo follows a single market event through the complete FinFlow pipeline:

1. **"Here's what happened"** -- Sources detected an ECB rate decision event (pre-seeded)
2. **"Here's what we suggest"** -- Command Center shows impact analysis, suggests EUR/USD report
3. **"Watch it work"** -- Pipeline Monitor streams FA/TA analysis, content generation, CTA generation in real time
4. **"You stay in control"** -- Content HITL: approve, edit, or reject the generated article
5. **"Now in your language"** -- Translation engine runs with the client's glossary and brand voice
6. **"Compliance is built in"** -- Compliance agent flags/fixes regulatory issues (MiFID II)
7. **"Final sign-off"** -- Compliance HITL: approve the compliance-modified version
8. **"Everywhere at once"** -- Publish Preview shows the same content adapted for Twitter, LinkedIn, email, blog, Telegram, learning portal -- all branded as the client's own
9. **"The full picture"** -- Report Viewer shows the finished product across languages and audience levels

## 3. Data Model

### Entity: DemoEvent (pre-seeded, read-only)

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| eventId | string | PK | Unique event identifier |
| headline | string | NOT NULL | Event headline |
| summary | string | NOT NULL | 2-3 sentence summary |
| sources | string[] | NOT NULL | Source tags (Reuters, Finnhub, etc.) |
| impactLevel | "high" \| "medium" \| "low" | NOT NULL | Impact classification |
| publishedAt | string | NOT NULL | ISO timestamp |
| instruments | DemoInstrument[] | NOT NULL | Affected instruments with suggested directions |
| relatedArticles | DemoArticle[] | NOT NULL | Source articles for the FA agent |

### Entity: DemoInstrument

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| symbol | string | NOT NULL | e.g. "EURUSD" |
| name | string | NOT NULL | e.g. "EUR/USD" |
| direction | "bullish" \| "bearish" \| "neutral" | NOT NULL | Suggested direction |
| confidence | number | NOT NULL, 0-100 | Impact confidence |

### Entity: DemoArticle

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| title | string | NOT NULL | Article headline |
| body | string | NOT NULL | Full article text for FA agent consumption |
| source | string | NOT NULL | Source name |
| publishedAt | string | NOT NULL | ISO timestamp |

### Entity: PipelineRun (shared type in `lib/types.ts`)

| Field | Type | Constraints | Description |
|-------|------|-------------|-------------|
| runId | string | PK | UUID |
| eventId | string | FK -> DemoEvent.eventId | Source event |
| instrument | string | NOT NULL | Selected instrument symbol |
| profileId | string | NOT NULL | Client profile to use |
| status | PipelineStatus | NOT NULL | Current pipeline status |
| stages | StageResult[] | NOT NULL | Ordered stage results |
| generatedContent | string \| null | | FA+identity output (pre-HITL) |
| translatedContent | string \| null | | Post-translation output |
| complianceResult | ComplianceResult \| null | | Compliance scan result |
| finalReport | DemoReport \| null | | Final assembled report |
| createdAt | string | NOT NULL | ISO timestamp |

```ts
type PipelineStatus =
  | "running"
  | "awaiting_content_approval"
  | "awaiting_compliance_approval"
  | "formatting_channels"
  | "completed"
  | "rejected";

type StageResult = {
  stage: string;
  status: "pending" | "running" | "complete" | "skipped" | "error";
  startedAt?: string;
  completedAt?: string;
  output?: unknown;
  tokens?: { input: number; output: number };
  costUsd?: number;
  agentName?: string;
  modelTier?: string;
};

type HitlDecision = {
  checkpoint: "content" | "compliance";
  action: "approve" | "reject" | "edit";
  decidedBy: string;
  decidedAt: string;
  editedContent?: string;
  dwellTimeMs: number; // how long reviewer spent before deciding
};

// --- Store interface (shared, not demo-specific) ---

interface PipelineRunStore {
  save(run: PipelineRun): Promise<void>;
  get(runId: string): Promise<PipelineRun | null>;
  list(filters?: { status?: string; profileId?: string; limit?: number }): Promise<PipelineRun[]>;
  updateStage(runId: string, stage: string, update: Partial<StageResult>): Promise<void>;
  recordHitlDecision(runId: string, checkpoint: string, decision: HitlDecision): Promise<void>;
};

type ComplianceResult = {
  jurisdiction: string;
  passed: boolean;
  modifications: ComplianceModification[];
  originalText: string;
  modifiedText: string;
};

type ComplianceModification = {
  type: "rewrite" | "addition" | "removal";
  description: string;
  original: string;
  replacement: string;
  rule: string; // e.g. "MiFID II Art. 44"
};

type DemoReport = {
  title: string;
  instrument: string;
  outlook: string;
  confidence: number;
  sections: ReportSection[];
  technicalData: Record<string, string | number>;
  scenarios: { label: string; probability: number; target: string; description: string }[];
  disclaimer: string;
  complianceBadge: { passed: boolean; jurisdiction: string; modificationCount: number };
  qualityScores: Record<string, number>; // from 13-metric scoring
  languages: { code: string; content: string }[];
  channelOutputs?: ChannelOutput[]; // populated after channel formatting stage
};

type ReportSection = {
  title: string;
  body: string;
};

type ChannelOutput = {
  channel: ChannelType;
  formattedContent: string;
  metadata: ChannelMetadata;
};

type ChannelType = "twitter" | "linkedin" | "instagram" | "blog" | "email" | "learning_portal" | "telegram";

type ChannelMetadata = {
  characterCount: number;
  characterLimit: number | null; // null = no limit
  truncated: boolean;
  hashtags?: string[];
  subject?: string; // email subject line
  previewText?: string; // email preview text
  cta?: { text: string; url: string };
  imagePrompt?: string; // suggested image description for channels that need visuals
};

type TenantTheme = {
  accentColor: string; // hex, e.g. "#2563eb"
  accentMuted?: string; // rgba override; auto-derived from accentColor if absent
  accentSubtle?: string; // rgba override; auto-derived from accentColor if absent
  logoUrl?: string; // URL to logo image; omit for text-only fallback
  companyName: string; // displayed in topbar, login, reports
  reportBranding?: {
    headerText?: string; // replaces "FinFlow" in report headers
    disclaimerOverride?: string; // replaces default disclaimer
    complianceJurisdiction?: string;
  };
  emailBranding?: {
    fromName?: string; // e.g. "Tier1FX Research"
    footerHtml?: string; // email footer with company details
  };
  socialHandles?: {
    twitter?: string; // e.g. "@tier1fx"
    linkedin?: string;
    instagram?: string;
    telegram?: string;
  };
};
```

> **Storage note:** `PipelineRun`, `PipelineStatus`, `StageResult`, `HitlDecision`, and `PipelineRunStore` live in shared `packages/api/src/lib/types.ts` (not `demo/types.ts`). Demo uses `InMemoryRunStore` (`Map<string, PipelineRun>`). Production swaps to `PostgresRunStore` (Drizzle) with no UI changes.

## 4. API Surface

All demo routes live under `/demo` to avoid collision with existing routes.

| Method | Path | Request Body | Response (200) | Errors | Auth | Purpose |
|--------|------|-------------|----------------|--------|------|---------|
| GET | /demo/events | -- | `{ events: DemoEvent[] }` | -- | none | List pre-seeded market events |
| GET | /demo/events/:eventId | -- | `{ event: DemoEvent }` | 404 | none | Get single event with articles |
| POST | /demo/runs | `{ eventId, instrument, profileId, language }` | `{ runId, streamUrl }` | 400, 404 | none | Start a pipeline run; returns SSE stream URL |
| GET | /demo/runs/:runId/stream | -- | SSE stream | 404 | none | Real-time pipeline event stream |
| GET | /demo/runs/:runId | -- | `{ run: PipelineRun }` | 404 | none | Get run state (for reconnect) |
| POST | /demo/runs/:runId/approve-content | `{ action: "approve" \| "reject" \| "edit", editedContent?: string }` | `{ status: "ok" }` | 400, 404, 409 | none | Content HITL decision |
| POST | /demo/runs/:runId/approve-compliance | `{ action: "approve" \| "reject" \| "edit", editedContent?: string }` | `{ status: "ok" }` | 400, 404, 409 | none | Compliance HITL decision |
| GET | /demo/runs/:runId/report | -- | `{ report: DemoReport }` | 404, 409 | none | Get final report (only after completion) |
| GET | /demo/runs/:runId/channels | -- | `{ channels: ChannelOutput[] }` | 404, 409 | none | Get channel-formatted outputs (only after channel formatting completes) |
| POST | /demo/runs/:runId/publish | `{ channels: ChannelType[] }` | `{ published: ChannelType[], message: string }` | 400, 404, 409 | none | Simulate publishing to selected channels (no real publish; returns success) |
| POST | /demo/profiles/extract | `{ samples: string[] }` | `{ profile: ClientProfileData }` | 400 | none | Thin wrapper around existing `/profiles/extract` |
| GET | /demo/profiles | -- | `{ profiles: ProfileSummary[] }` | -- | none | List available profiles (pre-seeded + extracted) |
| GET | /demo/runs | -- | `{ runs: PipelineRun[] }` | -- | none | List all runs; optional query params `?status=completed&limit=20&profileId=xxx` |
| GET | /demo/runs/:runId/timeline | -- | `{ stages: StageResult[], hitlDecisions: HitlDecision[], summary: { totalDuration: number, totalCost: number, totalTokens: number, qualityAvg: number } }` | 404 | none | Structured timeline data for the Run Detail page |

**SSE Event Types** (on `/demo/runs/:runId/stream`):

```ts
type DemoSseEvent =
  | { type: "stage_started"; stage: string; description: string }
  | { type: "stage_completed"; stage: string; duration: number; tokens?: { input: number; output: number }; costUsd?: number }
  | { type: "agent_output"; stage: string; agentName: string; model: string; content: string; tokens: { input: number; output: number } }
  | { type: "deliberation"; stage: string; round: number; fromAgent: string; toAgent: string; content: string }
  | { type: "content_ready"; content: string } // triggers HITL gate
  | { type: "translation_started"; language: string }
  | { type: "translation_completed"; language: string; qualityScores: Record<string, number> }
  | { type: "compliance_started"; jurisdiction: string }
  | { type: "compliance_completed"; modifications: ComplianceModification[] } // triggers HITL gate
  | { type: "channel_formatting_started"; channels: ChannelType[] }
  | { type: "channel_formatted"; channel: ChannelType; characterCount: number } // emitted per channel as each completes
  | { type: "channel_formatting_completed"; channelCount: number } // all channels done
  | { type: "pipeline_completed"; runId: string }
  | { type: "pipeline_error"; error: string }
  | { type: "hitl_waiting"; checkpoint: "content" | "compliance" };
```

## 6. New Backend Work

### 6.1 Demo Event Seeding (`packages/api/src/demo/seed.ts`)

Pre-seeded market events with full article bodies for FA agent consumption. Ships with 3 events:

1. **ECB Rate Decision** (high impact) -- EUR/USD bearish, EUR/GBP bearish, Gold bullish
2. **US Crude Inventories** (medium impact) -- WTI bullish, Brent bullish
3. **Japan GDP Revision** (low impact) -- USD/JPY neutral

Each event includes 3-5 `relatedArticles` with realistic financial news body text (500-800 words each) that the FA agent can analyze.

### 6.2 Demo Pipeline Runner (`packages/api/src/demo/pipeline-runner.ts`)

Orchestrates the full E2E pipeline for a demo run. Manages:

- Stage sequencing with SSE emission at each transition
- Two HITL gates (content + compliance) that suspend pipeline execution
- Integration with existing translation engine
- Integration with new compliance agent

**Stages 1-2 (News Scraping + Data Ingestion):** Simulated. Emit SSE events with pre-seeded data after a 1-2 second delay (feels real, shows the stage progression).

**Stages 3-4 (Core Analysis + Identity Adaptation):** Real LLM calls. Reuse the FA agent prompt from `packages/api/src/benchmark/uniqueness-poc/prompts/fa-agent.ts` and the identity system from `packages/api/src/benchmark/uniqueness-poc/prompts/identities/`. For the demo, use the `in-house-journalist` identity (produces journalism-style market columns).

**Stage 5 (CTA Generation):** New. A single Haiku call that generates 2-3 CTAs appropriate to the content (e.g., "Open a demo account", "Read our EUR/USD guide", "Subscribe for real-time alerts"). Takes the generated content + client profile as input.

**Stage 6 (Content Assembly):** Deterministic. Combines FA analysis + identity output + CTAs into a structured `DemoReport` shape. No LLM call.

**Stage 7 (Analyst Review / Content HITL):** Pipeline suspends. SSE emits `hitl_waiting` with `checkpoint: "content"`. Resumes when `POST /demo/runs/:runId/approve-content` is called.

**Stage 8 (Translation):** Real. Calls the existing translation engine (`packages/api/src/pipeline/translation-engine.ts`) with the approved content and the client's profile. Emits SSE events for each translation sub-stage (translate, score, gate, specialists, re-score).

**Stage 9 (Compliance):** New. See 6.3.

**Stage 10 (Final Approval / Compliance HITL):** Pipeline suspends again. SSE emits `hitl_waiting` with `checkpoint: "compliance"`. Resumes when `POST /demo/runs/:runId/approve-compliance` is called. On approval, proceeds to Stage 11.

**Stage 11 (Channel Formatting):** After compliance approval, the channel formatter (see 6.5) adapts the approved content for all target channels. SSE emits `channel_formatting_started`, then `channel_formatted` per channel, then `channel_formatting_completed`. The pipeline then assembles the final `DemoReport` (including `channelOutputs`) and emits `pipeline_completed`. The UI auto-navigates to the Publish Preview page.

### 6.3 Compliance Agent (`packages/api/src/demo/compliance-agent.ts`)

New LLM agent (Haiku) that scans translated content for regulatory compliance issues. Configured per-jurisdiction.

**Input:** translated content string, jurisdiction (e.g. "mifid2"), client profile compliance patterns

**Output via tool_use:**

    type: "rewrite" | "addition" | "removal";

### 6.5 Channel Formatter (`packages/api/src/demo/channel-formatter.ts`)

Adapts approved, compliance-checked content for each output channel. Uses a hybrid approach: one Haiku LLM call generates the adaptive content (tone shifts, CTA placement, hashtag generation), then deterministic post-processing enforces character limits, formatting rules, and template structure.

**Input:** approved content string, client profile (including `TenantTheme`), list of target channels

**Channel rules (deterministic constraints applied after LLM adaptation):**

| Channel | Char limit | Tone shift | Image needed | CTA placement | Special rules |
|---------|-----------|------------|-------------|---------------|---------------|
| Twitter/X | 280 | Punchy, news-hook | No (text only) | Inline link | Max 3 hashtags, no markdown |
| LinkedIn | 3000 | Professional, thought-leadership | Optional | Footer | Paragraph breaks, 3-5 hashtags |
| Instagram | 2200 | Accessible, visual-first | Required (prompt only) | Bio link reference | Heavy hashtags (up to 15), emoji OK |
| Blog | None | Full editorial | Required (prompt only) | Inline + footer | Full markdown with H2/H3, meta description |
| Email | None | Direct, scannable | Optional header | Primary CTA button | Subject line (60 chars), preview text (90 chars), unsubscribe footer |
| Learning Portal | None | Educational, structured | No | End-of-lesson | Sections with learning objectives, key takeaways, optional quiz Qs |
| Telegram | 4096 | Concise, action-oriented | No | Inline | Telegram markdown (bold, italic, links), no images |

The deterministic post-processor then: truncates to character limits (marking `truncated: true`), applies template wrappers (email HTML, Telegram markdown), inserts tenant branding from `TenantTheme` (social handles, email footer, blog byline), and assembles final `ChannelOutput[]`.

### 8.2 Dynamic Tenant Theming

The design system supports per-tenant re-skinning via CSS custom property swaps at runtime. When a tenant is selected (login or context switch), `ThemeProvider` updates `:root` CSS variables.

**What swaps per tenant:**
- `--accent` / `--accent-muted` / `--accent-subtle` -- derived from `TenantTheme.accentColor`. If `accentMuted`/`accentSubtle` are not provided, they are auto-derived by applying 70% and 12% opacity to the accent hex.
- Logo in `TopBar` and `LoginPage` -- if `TenantTheme.logoUrl` is set, renders `<img>` at 32px height; otherwise renders `companyName` in serif font.
- Company name in sidebar header, report headers, email template branding.
- Social handles in publish preview channel cards.

**ThemeProvider implementation:**
```tsx
// packages/web/src/components/theme/ThemeProvider.tsx
function ThemeProvider({ children }: { children: React.ReactNode }) {
  const { activeProfile } = useDemoContext();
  const theme = activeProfile?.theme;

  useEffect(() => {
    const root = document.documentElement;
    if (theme?.accentColor) {
      root.style.setProperty("--accent", theme.accentColor);
      root.style.setProperty("--accent-muted", theme.accentMuted ?? hexToRgba(theme.accentColor, 0.7));
      root.style.setProperty("--accent-subtle", theme.accentSubtle ?? hexToRgba(theme.accentColor, 0.12));
    } else {
      // Reset to defaults
      root.style.removeProperty("--accent");
      root.style.removeProperty("--accent-muted");
      root.style.removeProperty("--accent-subtle");
    }
  }, [theme]);

  return <>{children}</>;
}
```

**Key constraint:** All existing components already reference `var(--accent)` via Tailwind utilities (`bg-accent`, `text-accent`, `border-accent`). No component changes are needed for the color swap -- only the CSS variable values change. Components that reference the logo or company name read from `DemoContext.activeProfile.theme`.

**Adding a new prospect theme:** Add a `TenantTheme` object to their client profile in `seed/profiles.ts` (or extract one during onboarding). The entire demo re-skins automatically.

### 8.3 Design Principles

- **Restrained darkness:** Every surface earns its lightness. The background hierarchy (root -> app -> surface -> raised) creates depth without brightness. Never use white backgrounds.
- **Muted authority:** The teal accent is used sparingly -- active states, primary actions, brand highlights. It should feel like a quiet signal, not a shout. Overusing accent kills the premium feel.
- **Typographic hierarchy through family, not weight:** Headlines use Playfair Display (serif) at normal weight. Body uses Inter. Data uses JetBrains Mono. The font family switch signals importance more than bold/size changes.
- **Motion as confirmation, not decoration:** Animations confirm user actions (fadeUp on mount, hover lift on cards, pulse on live indicators). Never animate for spectacle alone. Use `cubic-bezier(0.16, 1, 0.3, 1)` for exits, `cubic-bezier(0.34, 1.56, 0.64, 1)` for spring effects.
- **Data density without clutter:** Financial UIs need to show a lot of numbers. Use the mono font, muted colors for labels, and tight spacing (4px grid). Let the data breathe through border hierarchy, not whitespace.

## 10. HITL Flow Detail

### 10.1 Content HITL (Checkpoint 1)

**Trigger:** Pipeline runner emits `content_ready` SSE event with the generated content.

**UI state:** Pipeline Monitor shows a `hitl-block` card with pulsing border. "Review Required" button navigates to `/pipeline/:runId/approve/content`.

**Approval screen shows:**
- Generated article content (rendered as formatted text, not raw)
- Article metadata: instrument, outlook, confidence, identity used
- Word count, estimated read time
- Three actions: Approve & Translate, Edit & Approve, Reject

**On approve:** Pipeline runner resumes at Stage 8 (Translation). SSE emits `stage_started` for Translation.

**On edit:** Pipeline runner uses the edited content as input to Stage 8 instead of the original.

**On reject:** Pipeline status set to "rejected". SSE emits `pipeline_error` with reason "Content rejected by reviewer".

### 10.2 Compliance HITL (Checkpoint 2)

**Trigger:** Pipeline runner emits `compliance_completed` SSE event with modifications list.

**UI state:** Same pattern as content HITL. Pipeline Monitor shows the gate; button navigates to `/pipeline/:runId/approve/compliance`.

**Approval screen shows:**
- Compliance modifications list (as shown in mockup: type badge, original -> replacement, rule reference)
- Summary: instrument, outlook, jurisdiction, languages
- Three actions: Approve & Continue, Request Corrections, Reject

**On approve:** Pipeline completes. Report is assembled. SSE emits `pipeline_completed`.

**On request corrections:** (v1 behavior: same as reject with a different label. Real correction loop is out of scope for demo.)

**On reject:** Pipeline status set to "rejected".

### 10.3 Translation Learning Loop (Simulated)

Not a real loop in the demo. When the user edits content at either HITL checkpoint, a subtle toast appears: "Correction saved -- FinFlow will learn from this edit for future translations." No actual learning occurs. This plants the seed of the value prop without requiring real implementation.

## 13. Constraints

- **No authentication.** Demo has no real auth. The login screen stores a tenant name in React state. This is deliberate -- adding auth adds demo friction.
- **No database.** All state is in-memory (`InMemoryRunStore` implementing `PipelineRunStore`). Server restart clears all runs. Profiles persist only in the in-memory `ProfileStore`. Production swaps to `PostgresRunStore` with no code changes above the store interface.
- **Single concurrent run.** The demo supports one pipeline run at a time. Starting a new run while one is active is allowed but the old run is abandoned.
- **API cost awareness.** A full demo run makes ~5 LLM calls (FA core, identity, CTA, translation, compliance). At Opus for FA + Sonnet for identity + Haiku for CTA/compliance, estimated cost per run: $1-3. The translation engine may add 2-4 more calls depending on quality loop rounds.
- **No real scanning.** Sources and events are pre-seeded. No real RSS/API polling occurs.
- **Backend runs on same port.** The Hono server serves both existing routes (`/translate`, `/profiles`) and demo routes (`/demo/*`). Frontend proxies via Vite config.
- **Bun runtime.** Both API and web use Bun. No Node compatibility requirements.

## 14. Out of Scope

| Excluded | Reason |
|----------|--------|
| Real authentication / JWT / sessions | Adds demo friction; no security risk for internal demo |
| Real data source scanning / RSS polling | Would require `@wfx/sources` package (Workstream B); pre-seeded data is sufficient |
| Multi-tenant comparison mode | Demo is single-tenant; the PoC playground already handles multi-tenant |
| Real audience-level content adaptation | Beginner/Intermediate/Professional tabs exist but show same content; real adaptation needs identity variants |
| Real chart rendering (TradingView) | Placeholder is sufficient; TradingView Lightweight Charts integration is a post-demo enhancement |
| Real translation learning loop | Toast message simulates; actual feedback loop is a production feature |
| PDF export | Button present with "Coming soon" toast |
| Real publishing to external channels | Demo generates channel previews and simulates publishing; actual API integration with Twitter/LinkedIn/etc. is `@wfx/publishers` (Workstream D) |
| Database persistence | In-memory is fine for demo sessions |
| Mobile responsiveness | Demo is presented on laptop/projector; responsive design is post-demo |
| Admin page real functionality | Static display only |
| Correction loop on compliance rejection | "Request Corrections" behaves same as reject in v1 |
